# Perfetto 系统级性能分析报告

## 一、概览

| 指标 | 数值 |
|------|------|
| 采集时长 | 3562ms (3.6s) |
| 帧数 | 85 |
| 目标帧率 | 30 FPS |
| 实际帧率 | 24.3 FPS |
| 平均帧耗时 | 41.14 ms |
| 中位数帧耗时 | 40.69 ms |
| Jank 次数 | 0 |
| BigJank 次数 | 0 |
| 设备 CPU | 2 大核 (max 2035MHz) + 6 小核 (max 1804MHz) |

## 二、核心结论

> 实际帧率 24.3 FPS，未达 30 FPS 目标（差距 19%）。主要瓶颈为 **CPU-bound（主线程计算过重）**，叠加 **渲染线程调度不合理**（81.4% 时间在小核上运行）和 **确认的热降频**（thermalScore=5）。帧时分布相对均匀（无 Jank），说明问题是**稳态性能不足**而非偶发卡顿。

## 三、CPU 调度分析

### 大小核使用

| 线程 | 大核占比 | 小核占比 | 核迁移次数 | 迁移频率 |
|------|---------|---------|-----------|---------|
| UnityMain | **100%** | 0% | 89 | 25次/秒 |
| UnityGfxRenderS | **18.6%** | 81.4% | 475 | 133次/秒 |

- 主线程：100% 在大核（core 6/7），调度合理
- 渲染线程：**严重问题** — 仅 18.6% 在大核，81.4% 被调度到小核（主要在 core 0/1/3）
- 渲染线程核迁移 475 次 / 3.56s = **133次/秒**，远超正常值（<5次/秒），调度极不稳定

### Runnable 等待

| 线程 | avg | max | p95 | 判定 |
|------|-----|-----|-----|------|
| UnityMain | 0.088ms | 1.638ms | 0.394ms | 正常 |
| UnityGfxRenderS | 0.262ms | 4.692ms | 1.004ms | 正常偏高 |

- 主线程 runnable 等待极低（0.088ms），调度响应快
- 渲染线程 p95 达 1.004ms，偶发延迟存在但未达严重程度

### 唤醒延迟

| 线程 | avg | max | p95 | 判定 |
|------|-----|-----|-----|------|
| UnityMain | 0.088ms | 1.638ms | 0.394ms | 正常 (<0.5ms) |
| UnityGfxRenderS | 0.262ms | 4.692ms | 1.004ms | 正常 (<0.5ms) |

唤醒延迟均在正常范围，不存在 CPU 过载导致的唤醒慢问题。

### 被抢占分析

| 线程 | 抢占次数 | 频率 |
|------|---------|------|
| UnityMain | 197 | 55次/秒 |
| UnityGfxRenderS | 305 | 86次/秒 |

- 渲染线程被抢占 305 次，主要与其频繁在小核上运行有关（小核上竞争更多）
- 结合系统干扰分析，bluetooth/surfaceflinger 是主要抢占源

### 调度问题判定

**渲染线程调度严重不合理。** 81.4% 时间运行在小核上，核迁移 133次/秒，导致渲染线程实际执行效率大打折扣。这是典型的 vivo 设备激进节能策略问题——EAS 将渲染线程视为非关键任务而分配到小核。

## 四、CPU 频率分析

### 频率概况

| 集群 | 平均频率 | 最大频率 | 占比 |
|------|---------|---------|------|
| 大核 (core 6/7) | 1785 MHz | 2035 MHz | **87.7%** |
| 小核 (core 0-5) | 1317 MHz | 1804 MHz | 73.0% |

大核平均频率未达 max（87.7%），说明存在频率受限。

### 降频检测

- 总降频事件：**24 次**（core 6: 12次，core 7: 12次）
- 降频模式：1651 MHz → 1536 MHz
- 时间分布：
  - t=1196ms：首次降频（2次，两核同步）
  - t=3072-3086ms：**密集降频**（22次集中在 14ms 内）

### 降频分类判定

**确认为热降频** (thermalScore = 5)

| 方法 | 检测结果 | 得分 |
|------|---------|------|
| 负载-频率背离 | **20次**高负载(>98%)时降频 | +3 |
| 频率上限锁定 | 未检测到（各窗口 max 仍可达 2035MHz） | 0 |
| Thermal Zone 温度 | 无温度数据（trace 未采集） | 0 |
| 全核同步降频 | **12次**大核集体降至 1536MHz | +2 |

**关键证据**:
- 负载-频率背离：在 CPU 利用率 >98% 时频率仍被压低至 1536MHz，这不是正常 DVFS 行为
- 全核同步：core 6 和 core 7 在相同时刻降到完全相同的频率（1536MHz），符合 cluster-level thermal cap 特征

### 热压力风险评估

thermalScore = 5（≥5 为确认热降频）。虽然采集时长仅 3.56s，但后段已出现密集降频。如果持续运行，热降频程度会进一步加剧。当前设备处于 **中等热压力状态**。

## 五、多线程协作分析

### Main-Render 并行度

| 指标 | 数值 |
|------|------|
| Main 总运行时间 | 3102.52 ms |
| Render 总运行时间 | 1526.63 ms |
| Render 等待时间 (Semaphore.WaitForSignal) | 1787.27 ms |
| Render 活跃时间 | 1526.63 ms |

- Render 等待时间 1787ms > Render 活跃时间 1526ms → **Render 大部分时间在等主线程**
- Render 空闲率 = 1787 / (1787+1526) = **53.9%** > 30% → 严重等待
- 说明主线程是帧时间的主要贡献者

### 帧瓶颈判定

**Main-bound（主线程 CPU 计算瓶颈）**

证据：
1. PlayerLoop 平均 41.14ms > 帧预算 33.33ms（超出 23.4%）
2. Render 线程超过一半时间在等主线程（Semaphore.WaitForSignal 1787ms）
3. 无 `Gfx.WaitForPresent` 出现在 top slices → 非 GPU-bound
4. 主线程 top 耗时：渲染命令准备 (URP.Render 11.11ms/帧) + 逻辑更新 (BehaviourUpdate 6.48ms) + LateUpdate (6.18ms) + UI (3.98ms)

### Job Worker 利用率

- Worker 数量：**0**（未检测到匹配的 Worker Thread / Job.Worker 线程名）
- 可能原因：该 trace 中 Job System 的 Worker 线程名不符合默认匹配模式，或 Burst Job 使用了不同的线程命名
- 注意：主线程 `WaitForJobGroupID` 总耗时 135.46ms（avg 0.13ms），说明 Job System 确实在使用中，但 Worker 线程未被识别

## 六、系统干扰分析

### 干扰进程 TOP 5

| 进程 | 次数 | 总耗时 | 占帧预算比 |
|------|------|--------|-----------|
| com.android.bluetooth | 85 | 92.39ms | **3.25%** |
| /system/bin/surfaceflinger | 165 | 31.77ms | 1.12% |
| sh | 26 | 15.76ms | 0.56% |
| com.vivo.pem | 19 | 15.47ms | 0.55% |
| system_server | 39 | 8.41ms | 0.30% |

帧预算基准：85帧 × 33.33ms = 2833ms

### 干扰严重度评估

**轻微干扰**

- 系统进程总干扰 = 92.39 + 31.77 + 15.76 + 15.47 + 8.41 = **164.0ms**（占总帧预算 5.8%）
- `com.android.bluetooth` 异常活跃（92ms），可能是蓝牙设备连接中持续扫描
- `com.vivo.pem`（vivo Power/Energy Manager）出现说明设备正在积极进行功耗管理，与热降频判定一致
- surfaceflinger 31.77ms / 165次 = 0.19ms/次，属正常合成开销

## 七、GPU 负载分析

### GPU 数据可用性

**本 trace 未采集 GPU 数据。** `gpuAnalysis.available = false`，`gpu_counter_track` 表存在但无频率/利用率相关数据。

### GPU-bound 判定

**非 GPU-bound** [低置信度]

间接证据：
- 主线程 top slices 中**未出现** `Gfx.WaitForPresent` → 主线程没有等 GPU
- Render 线程大量时间在等主线程（Semaphore.WaitForSignal 占 52%）→ 不是 Render 提交后等 GPU
- 主线程自身计算就已超帧预算（41.14ms > 33.33ms）→ CPU-bound 特征明确

建议：下次采集时开启 GPU counter (`perfetto --gpu`) 以获取直接 GPU 数据。

| 降频次数 | 0 | 2 | **22** |
| Jank 数 | 0 | 0 | 0 |

### 性能趋势判定

**未检测到明显性能退化模式。**

分析：
- 三段帧率差异 < 3%（24.1/24.7/24.1），性能非常稳定
- 后段降频次数激增（22次 vs 前段 0次），但帧率未明显下降
- 原因：降频幅度有限（1651→1536MHz，仅降 7%），且大核平均仍在 1804MHz 运行，尚未影响帧率
- **预判**：如果采集时间更长（>10s），热降频累积效应可能导致帧率进一步下降

这属于 **"无模式 + 帧率不达标"** 组合 → 稳态 CPU-bound，性能瓶颈非时间相关。

## 九、优化建议

### P0 — 渲染线程绑核（预期收益：+3~5 FPS）

- **目标问题**：渲染线程 81.4% 在小核运行，核迁移 133次/秒
- **具体方案**：
  - 通过 `android.os.Process.setThreadAffinityMask()` 或 Unity C++ 层 `pthread_setaffinity_np()` 将 UnityGfxRenderS 绑定到大核
  - 或使用 Android GameMode API 请求 PERFORMANCE 模式：`GameManager.setGameState(GameState.PERFORMANCE)`
  - vivo 设备可通过 `com.vivo.pem` 白名单添加游戏进程
- **预期收益**：渲染线程在大核执行可提升约 50% 单线程性能（2035MHz vs 1804MHz + 大核 IPC 优势）

### P1 — 主线程逻辑优化（预期收益：-5~8ms/帧）

- **目标问题**：PlayerLoop 41.14ms > 帧预算 33.33ms，需降低 ~8ms
- **具体方案**：
  - URP.Render 占 11.11ms/帧 → 检查 camera stack 数量（当前 170次/85帧=2 camera），尝试合并相机
  - BehaviourUpdate 6.48ms + LateBehaviourUpdate 6.18ms → Profile 具体 MonoBehaviour，考虑将 heavy update 移入 Job System
  - CS:AOE.LuaMgr 总 370ms → 优化 Lua 热点函数
  - UGUI.Rendering.UpdateBatches 3.98ms → 减少 Canvas 重建频率
- **预期收益**：优化渲染 Pass 和逻辑可降低 5-8ms，使帧率接近 30 FPS

### P2 — 热管理策略（预期收益：防止长时间性能退化）

- **目标问题**：thermalScore=5 确认热降频，后段密集降频
- **具体方案**：
  - 实现动态负载调节：监测频率下降时自动降低渲染质量
  - 使用 `PowerManager.getThermalStatus()` 监听热状态
  - 在热压力升高时自动切换低画质模式（关阴影/降分辨率）
  - 考虑限帧（30→25 FPS）以降低整体功耗和发热
- **预期收益**：防止长时间运行后帧率进一步下降

## 十、补充说明

### 数据局限性
- 采集时长仅 3.56s（85帧），样本量较小，热降频的长期趋势无法充分观察
- 无 GPU counter 数据，GPU 瓶颈判定依赖间接推断
- 无 thermal_zone 温度数据，热降频判定依赖负载-频率背离和全核同步
- Job Worker 线程未被识别，Job System 利用率数据缺失

### 建议下一步
1. 延长采集时间至 30-60s 以观察完整热降频趋势
2. 开启 GPU counter 采集：`perfetto --gpu` 或在 trace config 中添加 `gpu_counters`
3. 添加 `linux.sys_stats` 采集以获取 thermal_zone 温度数据
4. 确认 Job Worker 线程名称模式（可在 config.json 的 `jobWorker.namePatterns` 中调整）
5. 针对 P0 建议做 A/B 测试：绑核前后对比 Render 线程大核占比和帧率变化
