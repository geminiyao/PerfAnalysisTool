# Unity Profiler 性能分析报告评分表

> 评审人对照此表逐项打分（1~5），用于对比不同 Agent/LLM 产出的报告质量。

---

## 总分计算

```
总分 = A类得分 × 0.4 + B类得分 × 0.35 + C类得分 × 0.25
每类得分 = 该类所有子项分数的平均值
```

---

## A 类：数据准确性（权重 40%）

### A1. 概览数据正确性

> frameSummary 表格中的数字是否与 preprocess-result.json 一致

| 分数 | 标准 |
|------|------|
| 5 | 所有字段（总帧数、FPS、均值、中位数、最差帧、Jank次数等）完全正确 |
| 4 | 1~2 个字段有微小四舍五入差异（如 28.35 → 28.4） |
| 3 | 3~4 个字段错误或缺失 |
| 2 | 超过一半字段错误 |
| 1 | 大量数据与源数据不符，疑似编造 |

## B 类：分析质量（权重 35%）

### B1. 瓶颈定位准确性

> 是否正确识别了每个 Jank 帧和热点 marker 的真正瓶颈节点

| 分数 | 标准 |
|------|------|
| 5 | 正确识别所有 BigJank/Jank 的瓶颈节点，瓶颈类型判定正确 |
| 4 | 主要瓶颈（BigJank）正确，个别次要问题定位有偏差 |
| 3 | BigJank 正确但一般 Jank/热点有遗漏或错误 |
| 2 | 主要瓶颈也有定位错误 |
| 1 | 瓶颈分析严重错误或指向了无关节点 |

**评分**: _____ / 5

### B2. 根因推理深度

> 是否结合项目知识库（unity-cpu-knowledge.md）和引擎知识给出了深层根因

| 分数 | 标准 |
|------|------|
| 5 | 结合知识库已知问题 + Unity 引擎知识 + 调用模式给出深层解释 |
| 4 | 有合理的根因分析，但未完全关联到项目已知问题 |
| 3 | 仅描述现象 + 简单归因（如"调用次数太多"） |
| 2 | 只有表面描述，无因果推理 |
| 1 | 无根因分析或分析完全错误 |

- [ ] 是否提及 AOE 项目的 MapSignificanceMgr 已知热点?
- [ ] 是否识别 GPU Bound / CPU Bound 的正确模式?
- [ ] 是否结合了 xLua 桥接、网络解包等项目特有知识?

## C 类：实用价值（权重 25%）

### C1. 优化建议可执行性

> 每条优化建议是否给出了可直接执行的具体步骤

| 分数 | 标准 |
|------|------|
| 5 | 每条建议有：具体操作步骤 + 目标代码/设置路径 + 预期收益 + 风险说明 |
| 4 | 多数建议具体，个别偏笼统但方向正确 |
| 3 | 有建议方向但缺少具体步骤（如仅说"分帧处理"但没说怎么做） |
| 2 | 空泛建议（如"优化性能"、"减少开销"、"建议优化"） |
| 1 | 无建议或完全不可执行 |

**反面示例** (不应出现):
- ❌ "建议优化物理性能"
- ❌ "减少网络开销"
- ❌ "考虑分帧处理"

**正面示例** (应该有):
- ✅ "在 ProcessTasks 循环内添加帧时间预算检查，每帧最多处理 N 个实体"
- ✅ "在 Edit → Project Settings → Quality 中将 Shadow Distance 从 X 降至 Y"
