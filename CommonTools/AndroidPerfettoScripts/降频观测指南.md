### 采集什么数据

| 数据 | 采集工具 | 用途 |
|------|---------|------|
| `.pftrace` | `record_tmaoe_thermal.bat` | 帧率/调度/频率/线程全量分析 |
| `scaling_max_freq` | 同上（adb sysfs） | **降频确认**: 系统是否限制了频率上限 |
| `cpuinfo_max_freq` | 同上 | 理论最大频率（基准） |
| `thermal_zone/temp` | 同上 | 设备温度 |
| `cooling_device/cur_state` | 同上 | thermal governor 是否激活 |
| 频率+温度 CSV | `monitor_freq.bat` | 长时间降频趋势曲线 |

### 降频判定（两级）

| 级别 | 判定方式 | 报告标注 |
|------|---------|---------|
| **确认** | `scaling_max_freq < cpuinfo_max_freq` | `confirmed` |
| **确认** | `cooling_device state > 0` | `confirmed` |
| **推测** | 频率可达性 / 负载背离 / 全核同步 / 持续低频 | `suspected [推测]` |

### 产出报告

| 报告 | 路径 | 内容 |
|------|------|------|
| Perfetto 分析报告 | `output/perfetto/perfetto-report_*.md` | A-H 八维度完整分析 |
| 频率监控 CSV | `output/freq_monitor_*.csv` | 频率+温度时间线（可画图） |
| 降频硬件快照 | `output/thermal_before/after_*.txt` | 采集前后 sysfs 对比 |

## 一、工具详细使用

### 1.1 record_tmaoe_thermal.bat — 一键采集（日常推荐）

**操作**: 连接手机 → 打开游戏 → 双击 bat → 等 10s → 看终端

**产出**:
```
output/
├── trace_20260511_1430.pftrace          ← 给 skill 分析
├── thermal_before_20260511_1430.txt     ← 采集前硬件状态
└── thermal_after_20260511_1430.txt      ← 采集后硬件状态
```

**看结果** — 终端直接显示，关键看这一行：
```
采集前: scaling_max_freq=2035000  cpuinfo_max_freq=2035000  ← 相等 = 未降频 ✅
采集后: scaling_max_freq=1536000  cpuinfo_max_freq=2035000  ← 不等 = 降频! ❌
```

**与 skill 配合**: 采集后直接说 "分析 output/trace_*.pftrace，目标帧率 30"，skill 自动读取 thermal txt，给出确认级判定。

### 1.2 monitor_freq.bat — 持续监控（长时间压测用）

**适用**: 跑 5-30 分钟压测，观察降频趋势，替代 PerfDog 频率曲线

**操作**: 双击 → 玩游戏 → `Ctrl+C` 停止 → 看 CSV

**CSV 示例**:
```csv
timestamp,cpu6_cur,cpu6_max,...,thermal_zone0_temp,scaling_limited
1715400000,2035000,2035000,...,38200,NO       ← 正常
1715400010,1804000,1804000,...,42100,NO       ← 开始发热
1715400020,1536000,1536000,...,45300,YES      ← 降频了!
```

**快速看**: `scaling_limited` 列出现 `YES` = 正在降频

**画图** (Python):
```python
import pandas as pd, matplotlib.pyplot as plt

df = pd.read_csv("freq_monitor_*.csv")
df['sec'] = df['timestamp'] - df['timestamp'].iloc[0]

fig, (ax1, ax2) = plt.subplots(2, 1, figsize=(12, 6), sharex=True)
ax1.plot(df['sec'], df['cpu6_cur']/1000, label='实际频率')
ax1.plot(df['sec'], df['cpu6_max']/1000, '--', label='允许上限')
ax1.set_ylabel('MHz'); ax1.legend(); ax1.set_title('大核频率')

ax2.plot(df['sec'], df['thermal_zone0_temp']/1000, 'r')
ax2.axhline(y=42, color='orange', linestyle='--')
ax2.set_ylabel('°C'); ax2.set_xlabel('秒'); ax2.set_title('温度')
plt.tight_layout(); plt.show()
```

### 1.3 adb 一行命令 — 紧急检查（30 秒出结果）

```bash
# 是否降频？两列不同 = 在降频
adb shell "paste /sys/devices/system/cpu/cpu*/cpufreq/cpuinfo_max_freq /sys/devices/system/cpu/cpu*/cpufreq/scaling_max_freq"

# 温度多少？>42000 = 开始热了
adb shell "cat /sys/class/thermal/thermal_zone*/temp"

# cooling device 激活了吗？state>0 = 在限频
adb shell "for cd in /sys/class/thermal/cooling_device*; do echo $(basename $cd) type=$(cat $cd/type 2>/dev/null) state=$(cat $cd/cur_state 2>/dev/null); done"
```

## 二、降频判定方法

### 确认级（硬件状态 — 最权威）

| 方法 | 命令/数据 | 判定 |
|------|----------|------|
| 频率上限对比 | `scaling_max_freq` vs `cpuinfo_max_freq` | 不等 = **确认降频** |
| Cooling device | `cooling_device/cur_state` | > 0 = **thermal 已激活** |
| ADPF API | `AThermal_getCurrentThermalStatus()` | ≥ MODERATE = **确认降频**（需游戏集成） |

### 推测级（Perfetto 频率推断 — 标注 [推测]）

| 方法 | 判定条件 | 科学来源 |
|------|---------|---------|
| 频率可达性 | 实际最高频 < 理论 max 的 95% | 逻辑推导 |
| 负载-频率背离 | CPU 高负载时频率反降 | [Perfetto 官方](https://perfetto.dev/docs/analysis/cpu) + ARM IPA |
| 全核同步降频 | cluster 所有核心降到同一低频 | ARM big.LITTLE 架构原理 |
| 持续低频 | >30% 时间在 <80% max | Perfetto 社区 |
| 频率上限锁定 | 各时间窗口 max 频率递减 | 逻辑推导 |

### 对比：确认 vs 推测

```
确认级:  "系统明确在限频" — scaling_max_freq 被内核压低了
推测级:  "频率看起来不正常" — 但可能是 DVFS 正常调节

没有 sysfs 数据时，报告只给推测结论，标注 [推测]
有 sysfs 数据时，以硬件状态为准
```

```

   adb shell "paste .../cpuinfo_max_freq .../scaling_max_freq"

```

## 四、常见问题

**Q: cooling_device 读不到？**
→ 部分厂商限制非 root。用 `scaling_max vs cpuinfo_max` 替代，一样权威。

**Q: Perfetto 里看不出降频？**
→ Perfetto 记的是 `scaling_cur_freq`（实际频率），不是 `scaling_max_freq`（限制上限）。低负载时即使被限频，实际频率可能本来就低。**一定要用 sysfs 对比。**

**Q: PerfDog 显示频率低，Perfetto 正常？**
→ 时间点不同。降频是跑久了才出现的。用 `monitor_freq.bat` 找到降频开始时刻，再用 `record_tmaoe_thermal.bat` 在那个时刻抓 trace。

**Q: 怎么让游戏自动感知降频？**
→ 集成 ADPF: `AThermal_getThermalHeadroom(tm, 10) >= 1.0` = 正在降频。[文档](https://developer.android.com/games/optimize/adpf/thermal)
