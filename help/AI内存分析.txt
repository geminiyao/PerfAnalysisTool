三、Harness Engineering：驯服 AI 的不确定性
最近 Harness Engineering（驯服工程）在 Agent 社区很火——核心理念是：AI Agent 不像传统程序那样按部就班，它会自己做决策、调用工具、处理意外。这种灵活性是优势，但也带来了不确定性。Harness Engineering 就是用朴素的工程方法来"驯服"这种不确定性，让 Agent 从"能跑的 Demo"变成"可靠的产品"。

你可能会问：把 memgraph 直接丢给 ChatGPT 不就行了吗？我们最初也试过，结果发现三个问题：原始数据太大放不进上下文、LLM 对内存指标的理解经常出错、每次分析结果不稳定。这正是 Harness Engineering 要解决的问题。

NexRay Agent 在设计之初就全面践行了 Harness Engineering 的方法论，下面展开几个关键设计：

3.1 多阶段任务拆分——像专家团队一样分工
Harness Engineering 的第一条原则：不要把所有事情交给一个 Agent，拆分任务、各司其职、代码控制流转。

一个资深性能专家分析内存问题时，脑中的流程大致是：先看宏观构成 → 定位异常区域 → 深入堆栈归因 → 最后写结论。NexRay Agent 把这个过程拆成了三个专业角色的协作：

角色	职责	类比
调度专家	理解你的问题，决定分析策略，编排流程	项目经理
内存分析专家	精通 iOS/Android 内存体系，负责数据分析和异常定位	领域专家
报告专家	将分析结论结构化，生成可读性强的报告	技术写作
每个角色有独立的知识库和精细化的工具权限——调度专家只能调度，不能直接操作分析工具；内存分析专家只能读取数据，不能生成最终报告。这种从代码层面而非 Prompt 层面做的强制约束，是 Harness Engineering 区别于"Prompt 调优"的关键。

NexRay Agent 采用两步法：

第一步（确定性脚本）：原始数据 → 结构化摘要 + 关键比率计算 + 异常标记
第二步（AI 推理）：  结构化摘要 → 深度分析 + 交叉验证 + 结论生成
第一步用脚本从几百 MB 的 memgraph 中提取关键指标、计算比率、标记异常——这些是确定性的，不依赖 LLM，保证数据准确。第二步才让 AI 基于结构化数据做推理判断。

同时，"内存分析专家"内置了深厚的 iOS 内存知识——XNU 内核 Footprint 精确计算公式、VMMap 多指标交叉分析方法（内存压力/碎片化/GPU 资源/虚拟地址浪费/僵尸内存/只读映射）等。这些知识不是通过 Prompt 临时注入的，而是系统性编码到 Agent 的行为模式中。

经验沉淀——"用得越多，越懂你的游戏"
Harness Engineering 的第三条原则：构建记忆机制，让 Agent 积累经验而不是每次从零开始。

传统 AI 工具每次对话都是"从零开始"。NexRay Agent 不一样——它会记住之前的分析经验：

短期记忆：当前会话中的分析上下文，支持多轮追问

长期记忆：每次分析的结论自动归档，下次分析同一个游戏时 Agent 会主动参考历史趋势

知识积累：从多次分析中自动提取"某个游戏的内存特征"——哪些区域通常占比大、哪些版本出过问题、哪些堆栈是常见热点

Agent 分析你的游戏越多，对它的"理解"就越深，给出的建议也越精准。就像一个长期跟你项目的性能专家，而不是每次都要从头介绍背景的外部顾问。

3.4 安全护栏——稳定可靠，不会"跑偏"
Harness Engineering 的第四条原则：给 Agent 装上护栏，用工程手段兜底 AI 的不确定性。

超时保护：单次分析设有时间上限，不会无限运行

循环检测：如果 Agent 反复执行相同操作，自动中止并报告

报告兜底：即使分析过程中出现异常，也能生成包含已有结论的报告

结果校验：关键数值通过脚本交叉验证，不完全依赖 LLM 的输出
