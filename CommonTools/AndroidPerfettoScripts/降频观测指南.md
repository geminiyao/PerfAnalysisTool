# Android CPU 降频观测 — 工具与使用指南

## 一、概述

本文档提供一套完整的 Android 设备 CPU 降频（thermal throttling）观测方案，涵盖：
- **采集工具**: 3 个 bat 脚本，覆盖不同使用场景
- **判定方法**: 确认级（硬件状态）+ 推测级（频率推断），分层给出结论
- **操作流程**: 从采集到分析的完整步骤

---

## 二、工具一览

| 工具 | 路径 | 用途 | 耗时 |
|------|------|------|------|
| `record_tmaoe.bat` | `CommonTools/AndroidPerfettoScripts/` | 原版 Perfetto 采集（仅 trace） | 10s |
| `record_tmaoe_thermal.bat` | `CommonTools/AndroidPerfettoScripts/` | **增强版**: Perfetto + 降频硬件状态 | 10s |
| `monitor_freq.bat` | `CommonTools/AndroidPerfettoScripts/` | **持续监控**: 每秒采集频率+温度 CSV | 持续 |

### 使用场景选择

| 场景 | 推荐工具 |
|------|---------|
| 日常性能采集 + 降频判定 | `record_tmaoe_thermal.bat` |
| 长时间跑游戏观察降频趋势 | `monitor_freq.bat` |
| 只看帧率/调度不关心降频 | `record_tmaoe.bat`（原版） |
| 复现问题时快速检查 | adb 一行命令（见下文） |

## 四、工具详细使用

### 4.1 record_tmaoe_thermal.bat — 一键采集（推荐日常使用）

#### 操作步骤

1. 手机连接 USB，打开游戏进入目标场景
2. 双击 `record_tmaoe_thermal.bat`
3. 等待 10 秒（期间在手机上操作游戏）
4. 采集完成后查看终端输出

#### 产出文件

```
output/
├── trace_20260511_1430.pftrace          ← Perfetto trace（给 skill 分析用）
├── thermal_before_20260511_1430.txt     ← 采集前硬件状态
└── thermal_after_20260511_1430.txt      ← 采集后硬件状态
```

#### 如何看结果

终端会直接显示对比结果。**关键看这一行**：

```
--- 采集前 频率上限 ---
scaling_max_freq: 2035000    cpuinfo_max_freq: 2035000   ← 相等 = 未降频
                                                    
--- 采集后 频率上限 ---
scaling_max_freq: 1536000    cpuinfo_max_freq: 2035000   ← 不等 = 正在降频!
```

**判定规则**：
- `scaling_max_freq == cpuinfo_max_freq` → ✅ **未降频**
- `scaling_max_freq < cpuinfo_max_freq` → ❌ **确认降频**，降幅 = (cpuinfo - scaling) / cpuinfo

#### 与 Perfetto Skill 配合

采集完成后，直接对 trace 文件调用 skill：
```
分析 output/trace_20260511_1430.pftrace，目标帧率 30
```

skill 会自动检测同目录下的 `thermal_before/after.txt`，在报告中给出 **确认级** 降频判定。

### 4.2 monitor_freq.bat — 持续频率监控

#### 适用场景

- 跑长时间压测（5-30 分钟），观察降频随时间的变化趋势
- 复现"跑久了帧率下降"问题时，同步记录频率和温度
- 替代 PerfDog 的 CPU 频率曲线功能

#### 操作步骤

1. 手机连接 USB
2. 双击 `monitor_freq.bat`（默认每秒采集一次）
3. 在手机上正常玩游戏
4. 完成后按 `Ctrl+C` 停止
5. 打开产出的 CSV 文件分析

#### 产出文件

```
output/freq_monitor_20260511_1430.csv
```

CSV 格式：
```csv
timestamp,cpu0_cur,cpu0_max,cpu4_cur,cpu4_max,cpu6_cur,cpu6_max,cpu7_cur,cpu7_max,thermal_zone0_temp,thermal_zone1_temp,scaling_limited
1715400000,1804000,1804000,1804000,1804000,2035000,2035000,2035000,2035000,38200,36100,NO
1715400010,1200000,1804000,1200000,1804000,1804000,1804000,1804000,1804000,42100,40500,NO
1715400020,1200000,1804000,1200000,1804000,1536000,1536000,1536000,1536000,45300,43200,YES
```

#### 各列含义

| 列 | 含义 | 单位 |
|-----|------|------|
| `timestamp` | Unix 时间戳 | 秒 |
| `cpu{N}_cur` | CPU N 当前实际频率 | KHz |
| `cpu{N}_max` | CPU N 当前允许的最大频率 (scaling_max_freq) | KHz |
| `thermal_zone{N}_temp` | 温度传感器 N 的读数 | 毫度 (÷1000 = °C) |
| `scaling_limited` | **是否正在限频** | YES/NO |

#### 如何分析 CSV

**快速判定**：看 `scaling_limited` 列，出现 `YES` 就是在降频。

**详细分析**（用 Excel 或 Python）：
```python
import pandas as pd
import matplotlib.pyplot as plt

df = pd.read_csv("freq_monitor_20260511_1430.csv")
df['time_sec'] = df['timestamp'] - df['timestamp'].iloc[0]
df['temp_C'] = df['thermal_zone0_temp'] / 1000

fig, (ax1, ax2, ax3) = plt.subplots(3, 1, figsize=(12, 8), sharex=True)

# 大核频率
ax1.plot(df['time_sec'], df['cpu6_cur']/1000, label='cpu6 实际频率')
ax1.plot(df['time_sec'], df['cpu6_max']/1000, label='cpu6 最大允许', linestyle='--')
ax1.set_ylabel('MHz')
ax1.legend()
ax1.set_title('大核频率')

# 温度
ax2.plot(df['time_sec'], df['temp_C'], color='red')
ax2.axhline(y=42, color='orange', linestyle='--', label='降频阈值参考 42°C')
ax2.set_ylabel('°C')
ax2.legend()
ax2.set_title('设备温度')

# 限频状态
ax3.fill_between(df['time_sec'], df['scaling_limited'].map({'YES':1,'NO':0}), alpha=0.3, color='red')
ax3.set_ylabel('限频')
ax3.set_xlabel('时间 (秒)')
ax3.set_title('是否限频')

plt.tight_layout()
plt.savefig('freq_analysis.png')
plt.show()
```

### 4.3 adb 一行命令 — 快速检查（最快）

#### 检查是否正在降频

```bash
adb shell "paste /sys/devices/system/cpu/cpu*/cpufreq/cpuinfo_max_freq /sys/devices/system/cpu/cpu*/cpufreq/scaling_max_freq"
```

输出示例（8 核设备）：
```
1804000  1804000    ← cpu0: 小核，未限频
1804000  1804000    ← cpu1: 小核，未限频
...
2035000  2035000    ← cpu6: 大核，未限频 ✅
2035000  1536000    ← cpu7: 大核，被限到1536! ❌
```

**两列数字不同 = 正在降频。**

#### 检查温度

```bash
adb shell "for tz in /sys/class/thermal/thermal_zone*; do echo $(basename $tz) $(cat $tz/type 2>/dev/null): $(cat $tz/temp 2>/dev/null); done"
```

输出示例：
```
thermal_zone0 cpu-0-0-usr: 42300    ← 42.3°C
thermal_zone5 gpuss-0-usr: 44100    ← GPU 44.1°C
thermal_zone9 battery: 35200        ← 电池 35.2°C
```

温度 > 42000（42°C）通常开始触发降频。

#### 检查 cooling device 状态

```bash
adb shell "for cd in /sys/class/thermal/cooling_device*; do echo $(basename $cd) type=$(cat $cd/type 2>/dev/null) state=$(cat $cd/cur_state 2>/dev/null) max=$(cat $cd/max_state 2>/dev/null); done"
```

输出示例：
```
cooling_device0 type=thermal-cpufreq-0 state=0 max=15    ← 小核未限频
cooling_device1 type=thermal-cpufreq-1 state=3 max=10    ← 大核限频! state=3
cooling_device2 type=thermal-devfreq-0 state=0 max=5     ← GPU 未限频
```

`state > 0` = thermal governor 已激活限频，`state / max_state` = 限频比例。

## 五、降频判定方法总结

### 5.1 判定层级

```
判定结果:
├── confirmed (确认降频)
│   ├── scaling_max_freq < cpuinfo_max_freq     [sysfs，最权威]
│   ├── cooling_device cur_state > 0            [sysfs]
│   └── AThermal status >= MODERATE             [ADPF API，需游戏集成]
│
├── suspected (疑似降频) [仅 Perfetto 数据时使用，标注 [推测]]
│   ├── 频率可达性: 实际最高频 < 理论最高频的 95%
│   ├── 负载-频率背离: CPU 高负载时频率仍在下降
│   ├── 持续低频: >30% 时间运行在 <80% 最大频率
│   ├── 全核同步降频: 同 cluster 所有核心降到同一低频
│   └── 频率上限锁定: 各时间窗口的 max 频率在递减
│
└── none (未检测到降频)
```

### 5.2 推测级方法的科学来源

| 方法 | 来源 | 原理 |
|------|------|------|
| 频率突降且持续不回升 | [Perfetto 官方分析指南](https://perfetto.dev/docs/analysis/cpu) | "Sudden, sustained frequency reductions" |
| 高负载时频率下降 | Perfetto 社区 + [ARM IPA 技术文档](https://blog.csdn.net/sinat_32960911/article/details/132192629) | 负载高时 DVFS 应该升频，不升反降 = 被限制 |
| 全核同步降频 | ARM big.LITTLE 架构 | 同一 cluster 共享频率域，thermal cap 作用于 cluster |
| 频率可达性 | 逻辑推导（无直接文献） | 如果 trace 期间频率从未达到理论 max，说明可能被限制 |

### 5.3 确认级方法的详细说明

参考: [thermal-throttling-reference.md](./references/thermal-throttling-reference.md)

## 六、完整操作流程

### 场景 A: 日常性能采集（推荐）

```
1. 连接手机，打开游戏进入场景
2. 双击 record_tmaoe_thermal.bat
3. 等待 10s，操作游戏
4. 查看终端：scaling_max_freq vs cpuinfo_max_freq
   → 不等 = 确认降频
   → 相等 = 未降频
5. 将 .pftrace 文件交给 skill 分析
   → 报告中 throttleVerdict.level = confirmed / suspected / none
```

### 场景 B: 长时间压测（观察降频趋势）

```
1. 连接手机，打开游戏
2. 终端1: 双击 monitor_freq.bat（持续监控）
3. 终端2: 双击 record_tmaoe_thermal.bat（可选，在关键时刻采集 trace）
4. 玩游戏 5-30 分钟
5. Ctrl+C 停止 monitor_freq
6. 打开 CSV，用 Excel 或 Python 画图
   → scaling_limited 列出现 YES 的时刻 = 降频开始时间
   → 温度曲线 = 设备热状态变化
```

### 场景 C: 紧急排查（最快）

```
1. 游戏正在运行，怀疑有降频
2. 开终端执行:
   adb shell "paste /sys/devices/system/cpu/cpu*/cpufreq/cpuinfo_max_freq /sys/devices/system/cpu/cpu*/cpufreq/scaling_max_freq"
3. 两列不同 = 确认降频
4. 可选: 同时看温度
   adb shell "cat /sys/class/thermal/thermal_zone*/temp"
```

## 七、常见问题

### Q: cooling_device 读不到怎么办？
部分厂商（华为/vivo）限制了非 root 访问。用 `scaling_max_freq vs cpuinfo_max_freq` 替代，效果相同。

### Q: thermal_zone 太多，看哪个？
一般看名称包含 `cpu` 的 zone（如 `cpu-0-0-usr`）。可以先 `adb shell "cat /sys/class/thermal/thermal_zone*/type"` 看类型。

### Q: 降频了但 Perfetto 里看不出来？
Perfetto 的 cpufreq counter 记录的是 `scaling_cur_freq`（实际运行频率），不是 `scaling_max_freq`（系统限制）。如果 CPU 负载低，即使被限频了，实际频率可能本来就低于限制值，表面上看不出区别。**所以一定要用 sysfs 对比 scaling_max vs cpuinfo_max，这才是确定性证据。**

### Q: PerfDog 看到频率很低，但 Perfetto 里频率正常？
可能时间点不同。降频是**动态的**——游戏跑久了设备发热才降频。Perfetto 采集的 10s 可能恰好在降频之前。**建议用 monitor_freq.bat 持续监控，找到降频开始的时间点，再用 record_tmaoe_thermal.bat 在那个时刻采集 trace。**

### Q: 如何让游戏自动感知降频？
集成 Android ADPF Thermal API:
```cpp
AThermalManager* tm = AThermal_acquireManager();
float headroom = AThermal_getThermalHeadroom(tm, 10);
if (headroom >= 1.0f) {
    // 正在重度降频，自动降画质
}
```
参考: [ADPF Thermal API Guide](https://developer.android.com/games/optimize/adpf/thermal)
